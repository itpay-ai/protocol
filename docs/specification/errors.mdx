# Errors

## Overview

The ItPay Protocol uses a consistent, structured error format across all API endpoints. Errors follow the conventions established by Stripe and other major payment APIs, making integration familiar and predictable.

All error responses use the appropriate HTTP status code and return a JSON body with a standard structure.

## Error Response Format

Every error response follows this structure:

```json
{
  "error": {
    "type": "invalid_request_error",
    "code": "parameter_missing",
    "message": "Missing required parameter: 'amount'.",
    "param": "amount",
    "doc_url": "https://docs.itpay.io/api/errors#parameter_missing",
    "request_id": "req_8Hk2mN4qR7wX9"
  }
}
```

### Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| `error.type` | string | High-level category of the error |
| `error.code` | string | Machine-readable error code |
| `error.message` | string | Human-readable description of what went wrong |
| `error.param` | string or null | The parameter that caused the error (if applicable) |
| `error.doc_url` | string | Link to documentation for this error code |
| `error.request_id` | string | Unique identifier for the request (use when contacting support) |

### Response Headers

| Header | Description |
|--------|-------------|
| `Itpay-Request-Id` | Same as `error.request_id` |
| `Itpay-Version` | The API version used for this request |
| `Itpay-Error-Type` | The error type for quick machine parsing |
| `Retry-After` | Present only for rate limit errors |

## Error Types

| Type | HTTP Status | Description |
|------|-------------|-------------|
| `api_error` | 500 Internal Server Error | An unexpected server-side error occurred. These are rare and should be reported. Retry with exponential backoff. |
| `invalid_request_error` | 400 Bad Request | The request was malformed or contained invalid parameters. Fix the request before retrying. |
| `authentication_error` | 401 Unauthorized | The API key was missing, invalid, or revoked. Check your authentication credentials. |
| `authorization_error` | 403 Forbidden | The API key does not have permission to perform the requested action. Check key scopes. |
| `not_found_error` | 404 Not Found | The requested resource does not exist. Check the resource ID. |
| `payment_error` | 402 Payment Required | The payment operation failed due to a payment-specific issue (insufficient funds, card declined, etc.). |
| `rate_limit_error` | 429 Too Many Requests | Too many requests were sent in a short period. Respect the `Retry-After` header and implement backoff. |

### Type Selection Logic

When integrating, use the error `type` for general flow control and the `code` for specific handling:

```python
def handle_api_error(error):
    if error["type"] == "payment_error":
        logger.warning(f"Payment failed: {error['code']}")
        notify_user_payment_failed(error)
    elif error["type"] == "rate_limit_error":
        retry_after = int(response.headers.get("Retry-After", 5))
        time.sleep(retry_after)
        retry_request()
    elif error["type"] == "api_error":
        logger.error(f"Server error: {error}")
        retry_with_backoff()
    else:
        logger.error(f"Unhandled error: {error}")
```

## Rate Limiting

### Rate Limit Algorithm

The ItPay Protocol uses a **token bucket** algorithm for rate limiting. Each API key has a bucket that refills at a steady rate.

### Rate Limits by Plan

| Plan | Requests per Second | Burst Capacity | Concurrent Requests |
|------|--------------------|----------------|---------------------|
| Sandbox | 25 | 50 | 10 |
| Starter | 100 | 200 | 25 |
| Business | 500 | 1,000 | 100 |
| Enterprise | Custom | Custom | Custom |

### Rate Limit Headers

Every API response includes rate limit information in the response headers:

| Header | Description | Example |
|--------|-------------|---------|
| `Itpay-Rate-Limit-Limit` | Maximum requests per second | `100` |
| `Itpay-Rate-Limit-Remaining` | Requests remaining in current window | `87` |
| `Itpay-Rate-Limit-Reset` | Unix timestamp when the bucket refills | `1717000010` |
| `Retry-After` | Seconds to wait before retrying (only on 429) | `3` |

### Handling Rate Limits

```python
import time
import requests

def make_api_request(url, headers):
    response = requests.get(url, headers=headers)

    if response.status_code == 429:
        retry_after = int(response.headers.get("Retry-After", 5))
        logger.warning(
            f"Rate limited. "
            f"Retry after {retry_after}s. "
            f"Remaining: {response.headers.get('Itpay-Rate-Limit-Remaining')}"
        )
        time.sleep(retry_after)
        return make_api_request(url, headers)  # Retry

    return response
```

### Rate Limiting Best Practices

1. **Monitor the headers**: Track `Itpay-Rate-Limit-Remaining` to anticipate limits before they are hit.
2. **Implement exponential backoff**: On `429` responses, sleep for `Retry-After` seconds plus jitter.
3. **Avoid polling**: Use webhooks instead of polling for status updates.
4. **Batch operations**: Use list endpoints with appropriate `limit` parameters instead of individual requests.
5. **Distribute requests**: If you have multiple services, use separate API keys to avoid shared rate limits.
6. **Cache responses**: Cache GET responses where appropriate to reduce request volume.

### Rate Limit Violation Example

```json
{
  "error": {
    "type": "rate_limit_error",
    "code": "rate_limit_exceeded",
```

## Best Practices for Error Handling

### Client-Side Error Handling

```python
import requests
from time import sleep
from random import uniform

def itpay_request(method, path, api_key, **kwargs):
    """Make an ItPay API request with automatic error handling."""
    headers = kwargs.pop("headers", {})
    headers["Authorization"] = f"Bearer {api_key}"

    max_retries = 3
    for attempt in range(max_retries):
        response = requests.request(method, path, headers=headers, **kwargs)

        if response.status_code in (200, 201):
            return response.json()

        error = response.json().get("error", {})

        # Rate limit — retry with backoff
        if error.get("type") == "rate_limit_error":
            retry_after = int(response.headers.get("Retry-After", 2 ** attempt))
            sleep(retry_after + uniform(0, 1))
            continue

        # Server error — retry with backoff
        if error.get("type") == "api_error" and attempt < max_retries - 1:
            sleep(2 ** attempt + uniform(0, 1))
            continue

        # All other errors — raise immediately
        raise ItPayError(
            type=error.get("type"),
            code=error.get("code"),
            message=error.get("message"),
            param=error.get("param"),
            request_id=error.get("request_id"),
        )

    raise ItPayError(type="api_error", code="max_retries_exceeded")

class ItPayError(Exception):
    def __init__(self, type, code, message, param=None, request_id=None):
        self.type = type
        self.code = code
        self.message = message
        self.param = param
        self.request_id = request_id
        super().__init__(message)
```

### Logging

Always log error details for debugging while being careful not to log sensitive data:

```python
import logging

logger = logging.getLogger("itpay")

def handle_error(error):
    logger.error(
        "ItPay API error",
        extra={
            "error_type": error.get("type"),
            "error_code": error.get("code"),
            "request_id": error.get("request_id"),
            # DO NOT log: api_key, full payload, customer PII
        }
    )
```
