# Specification

This specification defines the authoritative protocol requirements for the ItPay Protocol. It covers the core object model, state machines, channel adapter interface, payment lifecycle flows, webhook event schemas, and security requirements.

**The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.**

## Conformance

### Conformance Targets

This specification defines conformance for four implementation targets:

1. **Platform Implementation**: A server-side implementation that hosts the protocol API, manages state machines, enforces authorization, and dispatches webhooks.
2. **Payee Integration**: A client integration that consumes the Platform API to create ServiceManifests, query payment status, and handle webhook events.
3. **Channel Adapter**: A pluggable module that implements the `ChannelAdapter` interface for a specific payment rail.
4. **Payer Client**: A consumer-facing application (web, mobile, CLI) that initiates payments on behalf of a Payer.

A claim of protocol conformance MUST specify which target(s) the implementation addresses.

### Conformance Requirements

#### General

1. A conforming **Platform** implementation MUST implement all MUST-level requirements in this specification.
2. A conforming **Platform** implementation SHOULD implement all SHOULD-level requirements.
3. A conforming **Payee Integration** MUST correctly produce and consume all protocol objects as specified.
4. A conforming **Channel Adapter** MUST implement the `ChannelAdapter` interface defined in the Architecture section.
5. A conforming **Payer Client** MUST authenticate requests using the defined authentication mechanism.

#### Version Negotiation

1. The Platform MUST advertise its supported protocol versions via a `GET /protocol/versions` endpoint.
2. API requests MUST include an `ItPay-Version` header specifying the protocol version. If omitted, the Platform MUST respond with HTTP 400.
3. The Platform MAY support multiple protocol versions simultaneously but MUST serve each version exactly as specified.

#### Error Handling

1. All API responses MUST use standard HTTP status codes.
2. Error bodies MUST conform to the `ErrorResponse` schema:

```json
{
  "error": {
    "code": "invalid_state_transition",
    "message": "Cannot transition PaymentIntent from 'pending' to 'refunded' without first reaching 'settled'.",
    "details": {
      "object_type": "payment_intent",
      "object_id": "pi_abc123",
      "current_state": "pending",
      "requested_transition": "refund"
    },
    "request_id": "req_7yK2pQ4mR9"
  }
}
```

3. Error codes MUST be namespaced by domain: `payment.*`, `manifest.*`, `channel.*`, `auth.*`, `validation.*`.
4. The Platform MUST NOT disclose internal implementation details in error messages.

### Extensibility

This specification defines a core set of objects and flows that are required for interoperability. Implementations MAY extend the protocol in the following ways:

1. **Custom Channel Adapters**: A Platform MAY implement channel adapters beyond the standard set defined in this specification. Custom adapters MUST implement the `ChannelAdapter` interface and MUST be registered in the Platform's capability registry.
2. **Custom Fields**: All protocol objects MAY include a `metadata` map (key-value pairs with string values). Implementations MAY add custom fields to the `metadata` map. The Platform MUST preserve all metadata fields through state transitions and MUST include them in webhook payloads.
3. **Custom Webhook Topics**: A Platform MAY define custom webhook event types prefixed with `custom.`. These events MUST follow the standard webhook payload structure.

Extensions MUST NOT conflict with any MUST-level requirement. An implementation that adds extensions MUST still pass all conformance tests defined for the core specification.

### Implementation Requirements

#### Platform Implementations

A Platform implementation MUST provide:

1. **REST API** implementing all endpoints defined in the Payment Lifecycle section.
2. **State Machine Engine** that enforces legal state transitions for all core objects.
3. **Webhook Dispatcher** that delivers events with at-least-once semantics and retry with exponential backoff.
4. **Channel Adapter Registry** that loads and manages channel adapter plugins.
5. **Idempotency Store** that guarantees exactly-once semantics for a minimum of 24 hours.
6. **Audit Log** recording every state transition with timestamp, actor, previous state, and new state.

A Platform implementation MUST persist all protocol objects durably. In-memory-only implementations do not satisfy the durability requirement.

#### Payee Integration Requirements

A Payee integration MUST:

1. Generate and include `Idempotency-Key` headers on all mutation requests.
2. Verify webhook HMAC-SHA256 signatures before processing events.
3. Acknowledge webhook delivery with HTTP 200 within 5 seconds.
4. Implement idempotent processing of webhook events using the event `id` field.
5. Use TLS 1.3 or higher for all API communication.

A Payee integration SHOULD:

1. Implement a circuit breaker for Platform API calls with exponential backoff.
2. Maintain a local cache of ServiceManifest states with reasonable TTL.
3. Handle webhook replay by checking the event `created` timestamp for ordering.

#### Channel Adapter Requirements

A Channel Adapter implementation MUST:

1. Implement all methods of the `ChannelAdapter` interface.
2. Return typed error responses for all failure modes.
3. Emit webhook events for channel-specific state transitions.
4. Support idempotency for payment initiation requests.
5. Report channel availability via the `ChannelCapabilities` registry.

## Compliance Verification

### Conformance Test Suite

A formal conformance test suite is maintained at [https://github.com/itpay/protocol-conformance](https://github.com/itpay/protocol-conformance). The test suite covers:

1. **Object Validation**: All core objects are tested for required fields, type constraints, and default values.
2. **State Machine Transitions**: Every legal and illegal transition is tested for each object type.
3. **API Endpoint Coverage**: All REQUIRED endpoints are tested with valid and invalid inputs.
4. **Webhook Schema Compliance**: Webhook payloads are validated against the JSON Schema definitions.
5. **Security Requirements**: HMAC verification, idempotency, TLS requirements, and rate limiting are tested.
6. **Negative Testing**: Unauthorized access, invalid state transitions, malformed requests, and replay attacks are tested.

### Claiming Conformance

To claim conformance, an implementation MUST:

1. Pass 100% of MUST-level conformance tests.
2. Pass 100% of SHOULD-level conformance tests OR document deviations with justification.
3. Provide a conformance statement listing the specification version, implementation target, and any extensions or deviations.
4. Submit conformance test results to the ItPay Protocol conformance registry (optional but RECOMMENDED).
