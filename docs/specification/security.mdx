# Security

## Key Security Principles

The ItPay Protocol is architected around four foundational security principles that govern every aspect of the system design.

### User Consent and Control

Every transaction requires explicit, cryptographically verifiable user consent. No party — merchant, gateway, or plugin — can initiate a payment or access user data without the user's active authorization. Consent is expressed through signed payloads and scoped API tokens that cannot be repurposed beyond their intended operation.

- **Transaction consent**: Each payment intent must be signed by the user's wallet or authorized session key before processing.
- **Data consent**: Access to user billing data, purchase history, or identity information requires per-scope authorization with clear expiration.
- **Revocability**: Users may revoke consent at any time, invalidating outstanding tokens and pending operations.

### Data Minimization

The protocol collects and retains only the data strictly necessary to complete a payment or maintain the service. This limits exposure in the event of a breach and reduces compliance burden.

- Only the payment amount, destination address, and a reference ID are required for core payment processing.
- Optional metadata (invoice numbers, line items) is encrypted by default and never stored in plaintext on the server.
- Personal identifiable information (PII) such as names, emails, or shipping addresses are never stored by the protocol layer — they remain with the merchant's first-party systems.

### No Custodial Money Flow

The ItPay Protocol never holds user funds at any point in the transaction lifecycle. Value moves directly from the user's wallet or account to the merchant's designated payout address. The protocol acts purely as a messaging and verification layer.

- Funds flow directly from payer to payee through the underlying blockchain or payment rail.
- The protocol only verifies that the transaction occurred and that the correct amount was settled.
- No hot wallets, no pooled funds, no settlement risk at the protocol level.

### Encryption at Rest and in Transit

All data is encrypted using industry-standard algorithms at every stage of its lifecycle.

- **In transit**: TLS 1.3 is REQUIRED for all API and webhook communications. No cleartext HTTP is permitted for production traffic.
- **At rest**: All stored payloads, API keys, webhook secrets, and transaction records are encrypted using AES-256-GCM.
- **Ephemeral data**: In-memory processing buffers are zeroed after each operation.

## Authentication

### API Key-based Authentication (Server-side)

All API requests to the ItPay Protocol must be authenticated using a server-side API key. API keys are bearer tokens that identify the calling merchant or application.

**API Key Format**

```
ip_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
ip_test_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

- Keys begin with a prefix indicating the environment: `ip_live_` for production, `ip_test_` for test/sandbox.
- The remainder is a 32-byte cryptographically random value encoded in base62 (44 characters).
- Total key length: 49 characters.

**Key Creation**

API keys are generated through the ItPay Dashboard or via a POST to the `/v1/api_keys` endpoint. When a key is created, the full key is displayed exactly once. After that, only a masked version is stored in the database:

```
ip_live_xxxxxxxxxxxx...abcd
```

The plaintext key is irrecoverable — if lost, a new key must be generated and the old one revoked.

**Key Rotation**

Keys should be rotated on a regular schedule:

| Key Type | Recommended Rotation | Maximum Age |
|----------|---------------------|-------------|
| Standard API Key | Every 90 days | 180 days |
| Admin API Key | Every 30 days | 90 days |
| Webhook Signing Secret | Every 180 days | 365 days |

To rotate a key:
1. Generate a new key.
2. Update all consuming services with the new key.
3. Verify that all services are operating correctly with the new key.
4. Revoke the old key via the Dashboard or API.

During rotation windows, both the old and new keys may be accepted simultaneously for up to 24 hours to prevent downtime.

**Authentication Header**

```
Authorization: Bearer ip_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

### Webhook Signature Verification (HMAC-SHA256)

All webhook events emitted by the ItPay Protocol include an `Itpay-Signature` header that contains an HMAC-SHA256 signature of the webhook payload. Recipients MUST verify this signature before processing the event to ensure authenticity and integrity.

**Signature Algorithm**

```
signature = HMAC-SHA256(
    key    = webhook_signing_secret,
    data   = timestamp + "." + payload_body
)
```

Where:
- `webhook_signing_secret` is the shared secret configured in the ItPay Dashboard.
- `timestamp` is the Unix timestamp (seconds since epoch) of when the event was created, as a string.
- `payload_body` is the raw JSON body of the webhook POST request.
- `.` is the literal period separator.

The `Itpay-Signature` header format is:

```
Itpay-Signature: t=1717000000,v1=abc123def456ghi789...
```

- `t` is the timestamp.
- `v1` is the HMAC-SHA256 hex-encoded digest.

**Verification Code Example**

```python
import hmac
import hashlib
import json
from typing import Tuple, Optional

def verify_webhook_signature(
    payload: bytes,
    signature_header: str,
    secret: str
) -> Tuple[bool, Optional[str]]:
    """
    Verify an ItPay webhook signature.

    Args:
        payload: Raw request body as bytes.
        signature_header: Value of the Itpay-Signature header.
        secret: Your webhook signing secret.

    Returns:
        Tuple of (is_valid, error_message).
    """
    try:
        # Parse the signature header
        parts = {}
        for pair in signature_header.split(","):
            key, value = pair.strip().split("=", 1)
            parts[key] = value

        timestamp = parts.get("t")
        signature = parts.get("v1")

        if not timestamp or not signature:
            return False, "Missing timestamp or signature in header"

        # Recompute the signature
        signed_payload = f"{timestamp}.{payload.decode('utf-8')}"
        expected = hmac.new(
            key=secret.encode("utf-8"),
            msg=signed_payload.encode("utf-8"),
            digestmod=hashlib.sha256
        ).hexdigest()

        # Constant-time comparison to prevent timing attacks
        if not hmac.compare_digest(expected, signature):
            return False, "Signature mismatch"

        return True, None

    except Exception as e:
        return False, f"Verification error: {str(e)}"
```

**Tolerance Window**

- Accept timestamps within ±5 minutes of your server's clock to prevent replay attacks.
- Reject events with timestamps outside this window even if the signature is valid.

## Encryption

### TLS 1.3

TLS 1.3 is REQUIRED for ALL production API and webhook traffic.

- TLS 1.2 MUST NOT be used in production.
- Cipher suites: TLS_AES_256_GCM_SHA384 or TLS_CHACHA20_POLY1305_SHA256 only.
- Certificate validation MUST be performed on all client and server connections.
- mTLS is recommended for inter-service communication within the protocol infrastructure.

### Payload Encryption for Sensitive Fields

Certain fields in the API payload may contain sensitive customer data. The protocol supports field-level encryption using the payload encryption scheme.

**Encrypted Fields**

| Field | Object | Encryption Required |
|-------|--------|---------------------|
| `customer_name` | Customer | Recommended |
| `customer_email` | Customer | Recommended |
| `shipping_address` | Checkout | Recommended |
| `billing_details` | PaymentIntent | Recommended |
| `metadata` | Various | Optional |

**Encryption Scheme**

Sensitive fields are encrypted using AES-256-GCM with a randomly generated nonce (12 bytes). The encrypted value is base64-encoded and stored in a nested envelope format:

```json
{
  "encrypted_data": "base64encoded...",
  "nonce": "base64encoded...",
  "key_id": "key_abc123"
}
```

**Encryption Code Example**

```typescript
import { randomBytes, createCipheriv } from "node:crypto";
import { Buffer } from "node:buffer";

function encryptSensitiveField(
  plaintext: string,
  key: Buffer,
  keyId: string
): object {
  const nonce = randomBytes(12);
  const cipher = createCipheriv("aes-256-gcm", key, nonce);
  const encrypted = Buffer.concat([
    cipher.update(plaintext, "utf-8"),
    cipher.final(),
  ]);
  const tag = cipher.getAuthTag();
  return {
    encrypted_data: Buffer.concat([encrypted, tag]).toString("base64"),
    nonce: nonce.toString("base64"),
    key_id: keyId,
  };
}
```

### Key Management and Rotation Policy

| Key Type | Algorithm | Key Length | Rotation | Storage |
|----------|-----------|------------|----------|---------|
| API Key | Random (base62) | 256 bits | 90 days | Hashed (bcrypt) |
| Webhook Secret | Random | 256 bits | 180 days | Encrypted at rest |
| Payload Key | AES-256-GCM | 256 bits | 30 days | HSM or KMS |
| TLS Certificate | ECDSA P-384 | 384 bits | 1 year | HSM or KMS |

- Key escrow is NOT supported. Lost keys must be rotated.
- All cryptographic operations should use a hardware security module (HSM) or cloud KMS in production.
- Key usage is audited: every encryption, decryption, and key rotation event is logged.

## Implementation Checklist

### Production Readiness Items

- [ ] TLS 1.3 enforced at the load balancer and application level.
- [ ] Webhook signature verification implemented and tested for every event type.
- [ ] API keys stored using bcrypt and never logged or exposed in responses.
- [ ] Rate limiting configured with appropriate burst and sustained limits.
- [ ] CORS properly configured to restrict origins.
- [ ] Input validation with size limits on all endpoints.
- [ ] SQL injection and NoSQL injection protections (parameterized queries).
- [ ] Sensitive fields encrypted at rest.
- [ ] Audit logging enabled for all authentication and authorization events.
- [ ] Health check endpoint does not expose sensitive information.
- [ ] Secrets rotated before their maximum age.
- [ ] Security headers set: `Strict-Transport-Security`, `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`.

### Common Pitfalls

- **Storing raw API keys**: Always store hashed keys (bcrypt). If the database is compromised, raw keys must not be recoverable.
- **Skipping webhook verification**: Without HMAC verification, an attacker can forge webhook events. Always verify.
- **Ignoring timestamp checks**: Signature verification without timestamp tolerance checks enables replay attacks.
- **Logging sensitive data**: API keys, signatures, and payloads should never appear in logs.
- **Using weak cipher suites**: Only use TLS_AES_256_GCM_SHA384 or TLS_CHACHA20_POLY1305_SHA256.
- **Hardcoding secrets**: Use environment variables or a secrets manager, never hardcode.

### Compliance Notes

- **PCI DSS**: The ItPay Protocol is designed to reduce PCI DSS scope by never storing credit card numbers. If you process card payments, additional PCI DSS compliance may be required for your application.
- **GDPR**: Data minimization principles help with GDPR compliance. Ensure you have a lawful basis for processing personal data and honor data subject access requests.
- **CCPA**: Users have the right to know what personal data is collected and to request deletion. Implement data deletion endpoints.
- **SOC 2**: The protocol infrastructure is SOC 2 Type II certified. Merchants using the protocol may inherit some controls but should perform their own assessment.

## Threat Model

### STRIDE Analysis

| Threat | Description | Impact | Mitigation |
|--------|-------------|--------|------------|
| **Spoofing** | Attacker impersonates a merchant or user | Unauthorized transactions | API key authentication, webhook HMAC, mTLS for internal services |
| **Tampering** | Attacker modifies a payment payload or webhook event | Incorrect payment amounts, fraud | HMAC-SHA256 signatures, TLS in transit, payload encryption |
| **Repudiation** | Merchant claims they didn't receive a webhook they processed | Disputes, chargeback liability | Signed webhook IDs, audit logging with immutable records |
| **Information Disclosure** | Attacker intercepts API responses or database contents | Data breach, PII exposure | TLS 1.3, field-level encryption, database encryption at rest |
| **Denial of Service** | Attacker floods the API with requests | Service outage, payment processing halts | Rate limiting, IP-based throttling, WAF, DDoS protection |
| **Elevation of Privilege** | Low-privilege key used to access admin functionality | Full system compromise | Granular API key permissions, role-based access control (RBAC) |

### Additional Threats and Mitigations

| Threat | Description | Impact | Mitigation |
|--------|-------------|--------|------------|
| **Replay Attack** | Attacker resends a captured webhook event | Duplicate order processing | Timestamp tolerance window (5 min) and idempotency keys |
| **API Key Leak** | API key exposed in client code, logs, or Git | Any unauthorized action | Short-lived keys, automatic revocation on leak detection, credential scanning |
| **Man-in-the-Middle** | Attacker intercepts traffic between merchant and protocol | Data interception, tampering | TLS 1.3 with certificate pinning recommended |
| **Race Condition** | Concurrent webhook processing leads to double spend | Financial loss | Idempotency keys, optimistic concurrency control on state transitions |
| **Timing Attack** | Attacker measures response times to infer secret values | Key recovery | Constant-time comparison (hmac.compare_digest) for all cryptographic operations |
